Release notes v3.25
===================

Version 3.25 is the culmination of 3 months of work involving over 320
pull requests. It is propedeutic to the 2026 release of Global Hazard
and Risk Models; as such, it focuses on improving the performance and reducing
the memory occupation, as well as supporting the latest science.
Users valuing stability may want to stay with the LTS release instead
(currently at version 3.23.4).

The complete set of changes is listed in the changelog:

https://github.com/gem/oq-engine/blob/engine-3.25/debian/changelog

A summary is given below.

Global SES
----------

We improved the script generating the global SES file and now we
store information about the sources, so that it is possible
to determine the name of the source that generated a given rupture.

We now store the calculation parameters into an `oqparam` dataset,
so that it is possible to read the generated HDF5 file as as a regular
datastore, although without an associated calculation ID.

We note tha the ruptures are not filtered anymore when they are
generated (except for the minimum magnitude filter) and therefore you
will see all the possible ruptures generated by the sources, even
on locations far away from the given sites; however, they will be
filtered and discarded when performing the calculation.

This is convenient when running calculations on a region
containing multiple countries, because all countries will see the same
ruptures/events and comparisons will become possible.

We made it is possible to run multiple calculations starting from the
same SES with a single command, as in the following example:

$ oq engine --run job_Laos.ini job_Brunei.ini job_Malaysia.ini \
                  job_Cambodia.ini job_Myanmar.ini job_Singapore.ini \
                  job_Philippines.ini job_Thailand.ini \
                  job_Timor_Leste.ini job_Indonesia.ini \
                  job_Vietnam.ini --hc SES.hdf5

The risk calculations will be run in parallel if the --multi flag
is passed, sequentially otherwise.

We note that the provisional syntax recognized in the job.ini file

rupture_model_file = SES.hdf5

has been removed in favor of using `--hc`, since it avoids a special
case and makes calculations starting from a SES regular calculations.

Not only `--hc` has been extended to accept an HDF5 file instead of a
calculation ID, it also accepts a `job.ini` file and it that case it
simply runs it before running the child calculation.

Hazard and risk calculations starting from a SES has been hugely
optimized. The crucial achievement was an extreme optimization of the
reading of the ruptures. Even if you have tens of millions of
ruptures, only the ruptures around the sites of interests will be
used. In previous versions of the engine the filtering was so
inefficient that the strategy was not viable, while now it is nearly
instantaneous.

We also optimized the management of the site collection (only the
sites close to the ruptures are considered, and only those
are associated to the global site parameters in the SES file).

We fixed a bug in the association between event IDs and rupture IDs
in the `events` table that caused some confusion, even if it did not
affect the correctness of the results.

We fixed a bug causing the engine to compute the GMFs on the original
sites and not the provided sites, when given.

We refined the estimated computational weight for the rupture
to greatly reduce the number of slow tasks, even if more work
on that is expected in the future.

We reduced the data transfer by reading the site collection from
the datastore and transferring only the filtered site IDs.

We added a consistency check and if the effective investigation
time is different from the one in the parent calculation an error is
raised early.

There is now a warning when trying to compute `avg_gmf` with too many
sites, so that there is reason when a calculation seems to hang.
Also, for simplicity now we are computing avg_gmf only in absence of a
parent calculation.

Hazard: preclassical
--------------------

Before performing a classical PSHA calculation, the engine performs an
analysis of the source models in the so-called called *preclassical
phase*.  In this phase the engine estimates the computational weight
of the sources, a crucial information for the performance of the next
step of te calculation. The estimate was very much euristic and failed
in a few cases, resulting in slow tasks killing the performance.

Now the computational weight is directly proportional to the number of
contexts generated by each source. This is simpler and much
better than before (particularly for point sources), resulting in a
huge reduction of the slow tasks issue.

The speedup depends very much on the model and how many cores
you have: the more cores, the worse the slow task issue can be.
For instance for the USA model we are now 2.5 time faster
than in version 3.24 on a machine with 192 cores; in other
models or with less cores the improvement can be insignificant.

We changed the way the `CompositeSourceModel` is stored and read from
the datastore: this is an implementation detail, but it has an effect on
performance because it allows the classical calculator to read the
sources directly from the disk instead of transferring these in a less
efficient way.

In some models the new preclassical is much faster than before because
we moved the gridding of the point sources from the master node to the
workers, i.e. it is parallelized and up to N times faster if you have
N cores. The actual speedup may vary since the new parallelization
strategy used in the preclassical phase is based on the spawing of
subtasks, and cannot be easily predicted.

Setting `complex_fault_mesh_spacing` in the job.ini is now mandatory
in calculations with complex fault sources. Before, it was optional:
when missing the engine took the default used for `rupture_mesh_spacing`
(5 km) which is too small, resulting in calculations up to 10, 20 times
slower than needed.

Hazard: classical
-----------------

There was a major change in the task distribution strategy also
in the classical phase of the computation.
In most cases the engine generates more tasks than before,
since it uses an advanced subtask strategy: if a task is taking too
long to complete, it is automatically split in subtasks. The splitting
is fully automatic but the user can tune it by using the `split_time`
parameter in the job.ini. It should never be necessary to touch it
and currently it should be considered an internal parameter: it could
change or disappear in the future.

In order to support the USA model, we started supporting region-dependent
gsim logic trees in version 3.24. The feature is called internally "ilabel",
since you can enable it by adding an `ilabel` column in the
site model file, with an integer which is referenced in the
`site_labels` dictionary in the `job.ini` file, an example being:

  site_labels = {"Cascadia": 1, "LosAngeles": 2}

The feature is documented in the section "Site-dependent logic trees"
of the manual. It was experimental in version 3.24 and had a
restricted range of validity while now it should work in all cases,
including disaggregation calculations. However, region-dependent logic
trees are ignored in event based calculations, only the default logic
tree is used and there are no plans to extend the feature.

Supporting the USA model also required changing the way the engine
managed cluster sources: now all cluster groups are managed together
and since there are around ~400 groups we require ~400 times less
memory than before.

More in general, we now avoid keeping in memory large arrays (the so called
RateMaps) when not needed. That reduced the memory consumption even in
absence of cluster sources. Thanks to that, we were able to remove the
auto-tiling functionality that was used to reduce memory. You can
still use full tiling but you you have to specify `tiling=true` in the
job.ini explicitly.

We changed the algorithm for saving the rates for large computations:
now the temporary files (there is one for each relevant group and IMT) are
stored in the calculation directory $OQDATA/calc_XXX and not in
`custom_tmp`. As a consequence in SLURM clusters there is no need to
configure a scratch directory anymore.

Classical calculations now work also on servers without a graphic
display since the environment variable MPLBACKEND="Agg" is automatically
set when generating PNG pictures.

Hazard: postclassical
---------------------

The parallelization in the postclassical phase, where hazard
curves/maps and their statistics are generated, has been significantly
improved by setting the number of tasks generated to the number of
available cores. Also the "combine pmaps" operation has been optimized
and now the entire postclassical, in the most common case of computing the
means, is purely dominated by the time spent in reading the rates from
the disk. Computing the quantiles is significantly slower and memory
intensive since it requires computing all realizations for each site.

We now are gzipping the rates dataset in the datastore,
thus reducing the disk space occupation by approximately 3 times
and reducing the time required to read the rates by the same amount.

A minor optimization is that now we avoid initializing the logic
tree twice when using the --hc option to re-run the postclassical
phase.

When starting a postprocessing calculation with `--hc` the flag
`use_rates` was not being honored, i.e. only the one set in the parent
calculation was considered.

We added a new exporter `hmaps-stats` producing one file per return period
and statistics, thus avoiding nested fields in the CSV header. This is
convenient for the purpose of building the Global Hazard Map.

Calculations with few sites
---------------------------

We introduced the possibility to specify in the `job.ini` file
a `siteid` parameter associated to the `sites` parameter, used
to give short (up to 8 character) names to the coordinates. This is
akin to specify a `custom_site_id` column in the site model file, and
indeed the `siteid` ends up inside the `custom_site_id` field of the
site collection.

`siteid` strings are restricted to the URL-safe base64 alphabet, so
that they can be used in web applications. The convenience is that it
is sufficient to list the coordinates and the site parameters will be
automatically associated from the site model file or from the
parameters of the parent calculation, if any.

hazardlib: general
------------------

There was some general infrastructure work on the GSIM classes, at the
level of the MetaGSIM class. Now each GMPE instance has an attribute
`._toml` which is automatically set and used to compute the hash of
the GMPE.  It means that *all* GMPEs are hashable, therefore they can be
used as keys in dictionaries and they can be cached.  Before we were
relying on the user remembering to set the `_toml` attribute or
remembering to call the `valid.gsim` factory function.

We changed the implementation of interpolated tables, keyed by gsim,
magnitude and IMT, used in subclasses of `GMPETable`. Before the
interpolated tables where computed in the `__init__` method, however
that caused problems because the interpolated magnitudes were
hard to pass correctly to the underlying GMPEs (in case
of advanced GMPEs).  Now the interpolation happens in the `compute`
method (i.e. in the workers, not in the master node) but it is still
performed only once because it is cached.

Thanks to this change the GMPE `GmpeIndirectAvgSA` now works when the
underlying GMPE is a GMPETable subclass. We also fixed the edge
case where the job.ini file does not contain the IMT `AvgSA`.

The change also fixed a number of bugs in Conditional Ground Motion Models,
which are ModifiableGMPE instances depending on a dictionary of underlying
GMPEs, keyed by the Intensity Measure Type. Now they should work with
all kinds of underlying GMPEs.

As a consequence of the change, we have now a general mechanism for
managing GSIM class warnings that guarantees that they are displayed
only once even if the GSIM is instantiated multiple times.

We removed from hazardlib dozens of calls to the method `super().__init__()`
since noe there is no need to call it anymore in the vast majority of cases,
resulting in less coupling and simpler code.

We added a method `CompositeSourceModel.set_msparams` to initialize
properly multifault sources after reading the source models from
hazardlib (this is expensive and needed only for classical analysis,
not for event based ones).

We added a method `CompositeSourceModel.get_cmakers` returning
a `ContextMakerSequence` object that can be used to implement
custom versions of the classical PSHA calculator.

We extended `CompositeSourceModel.get_sources` to accept an `smr` index so that
advanced users can perform analysis one source model at the time.

Both new methods and the `smr` index are documented in the manual, in
the section "Reading the hazard sources programmatically". Moreover,
we extended the documentation about implementing advanced GSIMs,
including the ones using Machine Learning models.

Internally, all the logic about the `CompositeSourceModel` has been moved into
a new module `openquake.hazardlib.source_group`.

We added a method `SiteCollection.lower_res` to reduce the resolution
of a site collection by using Uber's `h3` library; this is used by the
engine when prefiltering the sources and the ruptures.

We fixed a bug about the conversion to geometric mean, which was
applied to all IMTs, including the non-supported ones, causing
ZeroDivision errors.

Sources with a `NegativeBinomialTOM` temporal occurrence model (used
in the New Zealand model) are now treated differently, allowing for a
simplification, but the user will not see any significant change in
the numbers and the performance.

As a consequence of the simplification, the classmethod
`ContextMaker.from_srcs(sources, sitecol)` now returns a context array
instead of a list of context arrays.

Finally, after several years of deprecation, we removed the method
`get_mean_and_stddevs` from all GMPEs. If you still have code calling the
old method, you should replace it with the function
`contexts.mean_stds(rup_ctx, gsim, imt, idx)` following the examples in
https://github.com/gem/oq-engine/pull/11194/changes

Risk
-----

Since starting a risk calculation from a stochastic event set stored into
an HDF5 file has become *extremely efficient*, the next generation of the Global
Risk Model will use this strategy. Starting from a SES means that the
ground motion fields are computed on the fly and never stored, thus
avoiding inefficiencies due to saving and reading large amount of data.
That strategy has been available in the engine for nearly 10 years
by specifying `calculation_mode = ebrisk`, but now it is the default,
so the `ebrisk` calculator has been removed.

For sake of the Global Risk Model and since it is useful in general we
added a new output "Average Losses By Taxonomy" (`avg_losses_by`)
aggregating the average losses. If the exposure contains a
`MACRO_TAXONOMY` field, it aggregates by it too, meaning that the
exporter will produce two files `avg_losses_by_-taxonomy.csv` and
`avg_losses_by_-MACRO_TAXONOMY.csv`.

Scenario calculations have been changed to consider only the sites
with assets around the rupture (within the maximum distance), except for
conditioned scenarios. Before the full site collection
was considered, thus requiring more memory and computation time.

We extended the `minimum_intensity` feature to also work for secondary
perils. For instance, setting in the job.ini

minimum_intensity = {'LiqProb': .02, 'LSE': .001}

will discard liquefactions probabilities below 2% and liquefaction
spatial extent below 0.1%. That makes secondary peril calculations
faster and requiring less disk space.

In infrastrusture calculations, we turned the hard-coded parameter
`max_nodes_network` into a job.ini parameter with default value 1000.

If the site parameters are more distant than ASSOC_DIST=8 km
from the sites we now raise an error and not a warning.

For scenario_risk calculations with quantiles and a single GSIM
the output "Aggregate Risk Statistics" was not visible. This is now fixed.

We fixed a bug in `classical_risk` in presence of nontrivial weights
in the taxonomy mapping file, reported by Lisa Jusufi in the OpenQuake
mailing list.

oq commands
-----------

The internal command `oq run` has been changed to support workflow
files (see https://github.com/gem/oq-engine/issues/11029). Those are
TOML files describing multiple calculations that should be performed
together. This is essential for the 2026 Global Hazard Model and
Global Risk Models since we want to be able to be able to run all the
hazard models in the mosaic or computing the risk profiles for all the
countries in the world by using a single configuration file. The
format is still experimental and internal but it is supposed to become
official in the near future.

`oq run` also accepts a `--cache` flag: when set to true
calculations that have been already performed are not repeated.
To determine if the calculation has been already performed the engine
looks at the checksum of the input files, which is stored in the
database. The feature is NOT enabled by defaults since it is
potentially dangerous: for instance a bug fix to a GMPE is a change in
the code, not in the input files, so using --cache=true will retrieve
the old (wrong) results. This is why the cache has to be enabled
manually only when the user knows that there were no significant
changes to the code. This is essential for the ergonomics of workflow
calculations in presence of errors, since you can easily relaunch
workflow without having to repeat successful calculations.

We extended `oq shell` to accept dotted names, so that it is easy
to call Python modules as scripts. An example is

$ oq shell openquake.engine.global_ses --help

to generate the global Stochastic Event Set.

It is now valid to pass a "prejob.ini" file to `--hazard-calculation-id`,
rather then simply an integer ID. So a command like

$ oq engine --run job.ini --hc prejob.ini

will perform two calculations: first, the one corresponding to `prejob.ini`
and then the one corresponding to `job.ini`, starting from the previous one.

NB: the old syntax

$ oq engine --run prejob.ini job.ini  # first prejob.ini and then job.ini

still works, but it is deprecated and in the future only the explicit
syntax with `--hc` might be accepted.

We extended the list of calculations generated by
`--list-hazard-calculations` and `--list-risk-calculations` to show
the calculation_mode too.

The command `oq purge` has been extended to remove failed calculations,
old calculations or orphan calculations, i.e. calc_XXX.hdf5 files
not referenced in the database (see  `oq purge --help`).

Finally, `oq plot` has been optimized by avoiding a costly buffer
around the mosaic/country geometries. We also improved the plotting
of event based ruptures in calculations starting from a SES.hdf5 file.

WebUI
-----

There were some fixes to the navigation bar which was not displaying
properly some elements in TOOLS_ONLY mode.

We added some filtering functionality to the page displaying the
calculations. For instance calling the URL
https://hostname/v1/calc/list?user_name_like=%test%
will return only the calculations of the users containing "test" in
the name (internally we are performing a LIKE query in the database).

When exporting multiple files as a single archive, the engine was
littering the temporary directory with zip files. This is now fixed.
Also, now only the specified `custom_tmp` directory is used, as it should.

We refactored the WebUI javascript and css code separating the logic into
multiple files and adding integration tests written in the Playwright
framework.

We added an endpoint `v1/calc/validate_ini` to validate a local .ini file,
to be used in the contexts of the project PAPERS.

We added and `<calc_id>/extract/exposure_by_location` endpoint,
to extract the exposure aggregated by asset locations.

Dozens of pull requests were made in the engine codebase to support
the AELO and OQImpact platforms. However, since those are private
platforms, the related improvements will not be listed here.

IT
---

In this release we dropped support for Python 3.10 and added
support for Python 3.13. That included upgranding numpy to version 2
and all the numpy-dependent libraries (including the geospatial
libraries). Upgrade pyproj to 3.7.1 to address CRS-related crash with 6.7.1

We removed several warnings caused by the library upgrade, while
some still remain.

We are now setting the PYTHONUTF8 mode on Windows to avoid
possible encoding errors.

Database migrations are now automatically performed without asking
for an annoying user confirmation.

We fixed a couple of issues in install.py. When installing with a flag
like --version=3.23 (without the patch number) we now get the
latest available patch from PyPI and not the first patch release
3.23.0. When installing from a branch (i.e. with --version=engine-3.23)
we now extract the requirements from the branch and not from master.

We changed install.py to not install the master demos but install the
version demos instead.

A user reported that the LOCKDOWN option in docker (when giving the
command `docker run -e LOCKDOWN=True openquake/engine`) was not
honored. This is fixed.

Some important .txt files in the GSIMs were not distributed in the
packaged version of the engine. Now they are. Same forthe .onnx and
.onnx.gz files required by some model.  Moreover now the Python
package for the engine includes the demos.
